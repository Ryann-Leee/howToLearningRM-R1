Policy Optimization, explained

PPO 与 GRPO：奖励如何变成模型参数更新

这页把 PPO 和 GRPO 放到同一条训练链路里讲：模型先生成，reward function 打分， advantage 判断这次生成比预期好还是差，logprob loss 决定提高还是降低这条生成路径的概率。

Rollout

模型生成

给定 prompt，policy 采样一条或多条 response。

Reward

奖励打分

reward function 给每条 response 一个分数。

Advantage

相对好坏

把 reward 减去 baseline，得到鼓励或惩罚方向。

Loss

加权概率

用 advantage 加权生成 token 的 logprob。

Update

反向传播

更新 policy 参数，让好路径更可能、坏路径更不可能。

PPO：用 Value Model 判断“比预期好多少”

PPO 的关键不是直接最大化 reward，而是把 reward 转成 advantage。 advantage 告诉模型：当前动作相对 baseline 是更好还是更差。

状态、动作、奖励在 LLM 中是什么

生成文本时，每个 token 都可以看作一步动作。

s_t = prompt + 已生成的前 t-1 个 token
a_t = 第 t 个生成 token
π(a_t | s_t) = 当前上下文下生成该 token 的概率
prompt 因为 客服 A 更 具体 <answer>[[A]]</answer>

Value Model 的角色

value model 不是把稀疏奖励变成真正的稠密奖励。它估计从当前状态继续走， 未来大概能拿多少总回报，用作 baseline。

V(s_t) ≈ E[未来总回报 | s_t]
A_t = R_t - V(s_t)

如果实际回报比 V(s_t) 高，A_t 为正；如果实际回报比预期低，A_t 为负。

PPO 一轮训练的逻辑图

1

Policy 生成 response

π_old 根据 prompt 采样一条文本。

2

Reward Model 打分

得到最终 reward 或 token-level reward。

3

Critic 估计 V

value model 预测每个状态的未来回报。

4

计算 Advantage

用 return 减掉 V，降低方差。

5

PPO Clipped Loss

更新 policy，同时限制更新步幅。

GRPO：不用 Value Model，改用组内相对比较

GRPO 的想法很干净：同一个 prompt 多采样几条 response，把它们放在一组里比较。 组内平均 reward 就成了 baseline。

组内 Advantage

对同一个 prompt 采样 G 条 response，得到 G 个 reward。

A_i = r_i - mean(r_1, ..., r_G)
A_i = (r_i - mean) / (std + ε)

第二个版本做了标准化，尺度更稳。判对但大家都判对时，鼓励会变小； 很难的题里唯一判对时，鼓励会变大。

为什么它适合推理模型

推理任务经常只有最终答案可验证。GRPO 不要求训练一个 value model， 只要同题多采样，就能知道哪条路径相对更好。

response 1: +1 response 2: +1 response 3: -1 response 4: -1 response 5: +1
GRPO 的 baseline 来自“同一个 prompt 下其它回答的平均表现”，PPO 的 baseline 通常来自 critic/value model。 两者都在回答同一个问题：这次动作到底比预期好多少？

Advantage 如何进入 Logprob Loss

reward 本身不可微，不能直接反传。可微的是模型对已生成 token 的概率。 所以 PPO/GRPO 用 advantage 去加权 logprob。

基础形式

policy_loss = - Σ_t A_t log π_θ(a_t | s_t)

A_t 为正时，增大 logprob 会让 loss 下降；A_t 为负时，降低 logprob 会让 loss 下降。

PPO 的 clipped ratio

ratio_t = exp(logπ_new - logπ_old)
L_t = min(ratio_t A_t, clip(ratio_t, 1-ε, 1+ε) A_t)

clip 防止一次更新过猛。它保留“鼓励好样本、压低坏样本”的方向，但限制步幅。

Token 级概率是如何取出来的

训练时把 rollout 生成的 response 再喂给当前模型。模型输出每个位置的 vocab 概率， 然后 gather 出实际生成 token 的概率。

logits = model(prompt + response).logits log_probs = log_softmax(logits) token_log_probs = gather(log_probs, generated_token_ids) loss = -mean(mask * advantage * token_log_probs)

PPO 与 GRPO 的核心区别

它们都是 policy optimization。不同点主要在 baseline/advantage 的来源， 以及是否需要训练 value model。

维度 PPO GRPO
Baseline 来源 value model / critic 预测 V(s_t) 同一 prompt 下多条 response 的组内平均 reward
Advantage A_t = R_t - V(s_t)，常配合 GAE A_i = (r_i - mean(group)) / std(group)
是否需要 Critic 通常需要，且要训练 value loss 通常不需要单独 value model
采样方式 可一题一答，也可多采样 依赖同题多采样形成 group
适合场景 通用 RLHF，尤其已有强 reward model 和 critic 训练流程 数学、代码、推理、可验证答案，以及希望省掉 critic 的场景
代价 多一个 value model 训练和稳定性问题 同一 prompt 需要多条 rollout，采样成本更高

放回当前 RM-R1 项目

当前项目是在用 GRPO 训练一个生成式 Reasoning Reward Model，让它作为 judge 判断 A/B 回答哪条更好。

训练数据

每条样本包含客户问题、客服 A、客服 B，以及 winner 标签，例如 model_a 或 model_b。

Policy 输出

模型生成推理过程，并在结尾输出 <answer>[[A]]</answer> 或 <answer>[[B]]</answer>。

Reward Function

只检查最后 80 个字符中最终 A/B 是否与 winner 一致。判对 +1，判错 -1。

GRPO 更新

同一个 judge prompt 采样多条 response，组内标准化 reward，鼓励判对路径，压低判错路径。

训练结果

得到一个会读两条回答、推理并给出偏好判断的 generative judge model。

最后把三句话带走

Reward 给方向

reward function 判断这条生成好不好，但它本身通常不可微。

Advantage 给力度

PPO 用 value baseline，GRPO 用 group baseline，目的都是判断比预期好多少。

Logprob 负责更新

模型通过提高或降低已生成 token 的概率，把奖励信号落实到参数上。
