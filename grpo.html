一个训练 batch 可以包含很多 prompt；GRPO 会在每个 prompt 内部分别成组，再分别算组内均值、标准差和 advantage。

Group 在 GRPO 里指同一个 prompt 下采样出来的一组 rollout。它不是整个 batch，也不能把不同题目的回答混在一起比较； 因为只有题目相同，reward 的高低才代表“这几条回答谁相对更好”。

4. Advantage：把 raw reward 变成组内相对优势

GRPO 不训练 value model。它直接把同一组 reward 标准化，得到每条 rollout 的相对优势。

先算组内统计量

mean = sum(reward) / 7
std = sqrt(mean((reward - mean)^2))
0.00mean(r_group)
0.00std(r_group)

再算每条 advantage

A_i = (r_i - mean(r_group)) / (std(r_group) + ε)

判对的 rollout 比组平均高，所以 A 为正；判错或格式错比组平均低，所以 A 为负。

代码对齐真实训练

TRL 风格 GRPO token loss：先逐行看这段代码在算什么

上面的手算只展开 policy surrogate。真实训练里还会把当前模型和 reference model 比较， 加一个 KL 惩罚项，避免 policy 为了拿 reward 一步漂得太远。下面函数是单个 token 的简化版； 实际 TRL/GRPO 训练会把这些量扩展成 batch × response_len 的张量，再乘 response mask 后求平均。

1. 输入都是 logprob 
logprob_theta = ln(0.6)
logprob_old   = ln(0.5)
logprob_ref   = ln(0.5)
advantage     = 1.0

训练代码通常直接拿 log probability，因为模型前向输出 logits 后会转成 token logprob；相除时用 log 差再 `exp`，数值更稳定。

2. old/new 概率比 
ratio = torch.exp(logprob_theta - logprob_old)

这等价于 `exp(ln(0.6)-ln(0.5)) = 0.6/0.5 = 1.2`，意思是当前模型把这个 token 的概率提高到 old policy 的 1.2 倍。

3. 两个 surrogate 
surr1 = ratio * advantage
surr2 = torch.clamp(ratio, 1.0 - eps, 1.0 + eps) * advantage

`surr1` 是不限制步幅的目标，`surr2` 是把 ratio 限在 `[0.8, 1.2]` 后的目标。它对应前面手算里的 clipped ratio。

4. 取更保守的目标 
policy_objective = torch.min(surr1, surr2)

正 advantage 时，取较小值可以防止好 token 的概率被一次性抬太多；负 advantage 时，clip 同样限制更新幅度。

5. ref KL 惩罚 
log_ratio_ref = logprob_theta - logprob_ref
kl_penalty = torch.exp(log_ratio_ref) - log_ratio_ref - 1.0

这里比较的是当前模型和 reference model。`exp(x)-x-1` 是 GRPO/DeepSeek Math 常用的低方差 KL 估计形式，用来惩罚离 ref 太远的 token。

6. 组合并取负号 
token_loss = -(policy_objective - beta * kl_penalty)

训练目标是“提高 clipped objective，同时扣掉 KL 惩罚”。PyTorch 优化器最小化 loss，所以最后取负号。

完整代码

把逐行解释合起来，就是单 token 的 GRPO loss

import torch

def compute_grpo_token_loss(logprob_theta, logprob_old, logprob_ref, advantage, beta=0.1, eps=0.2):
    """
    计算单个 Token 的 GRPO 损失
    
    参数:
    logprob_theta: 当前模型在该 token 的 log 概率 (对应 ln(0.6))
    logprob_old:   采样时旧模型在该 token 的 log 概率 (对应 ln(0.5))
    logprob_ref:   参考模型在该 token 的 log 概率 (对应 ln(0.5))
    advantage:     该 token 所在整个句子的组内相对优势 (对应 1.0)
    """
    
    # 1. 计算概率比值 Ratio (利用指数转换 log 的差值)
    # 等价于 exp(ln(0.6) - ln(0.5)) = 0.6 / 0.5 = 1.2
    ratio = torch.exp(logprob_theta - logprob_old)
    
    # 2. 计算 Clipped Surrogate Objective
    surr1 = ratio * advantage
    surr2 = torch.clamp(ratio, 1.0 - eps, 1.0 + eps) * advantage
    policy_objective = torch.min(surr1, surr2)
    
    # 3. 计算 KL 散度惩罚 (DeepSeek Math 里的无偏估计形式)
    # log_ratio_ref = ln(0.6/0.5)
    log_ratio_ref = logprob_theta - logprob_ref
    kl_penalty = torch.exp(log_ratio_ref) - log_ratio_ref - 1.0
    
    # 4. 组合最终目标函数并取反 (因为 PyTorch 的优化器是最小化 Loss)
    # 对应 -(1.2 - 0.1 * 0.018)
    token_loss = -(policy_objective - beta * kl_penalty)
    
    return token_loss
这段代码的单 token 结果还不是最终 batch loss。真实训练会先对所有 response token 做同样计算， 用 response mask 排除 prompt/pad，再在 token、rollout、mini-batch 维度上求平均。

6. 从 token-level 账本汇总到 rollout-level 账本

上一章展开了一条 rollout 内部的 token 计算。这一章回到 7 条 rollout：每条 rollout 都会做同样的 old/new policy 比较，再得到自己的 policy loss。

response 级简写

ratio_i = exp(logπ_new(response_i) - logπ_old(response_i))
objective_i = min(ratio_i * A_i, clip(ratio_i, 0.8, 1.2) * A_i)
policy_loss_i = -objective_i

这只是 token 级计算的压缩写法

表里的 old/new logprob 可以理解为整条 response token logprob 的汇总。它保留了“和 old policy 比较、再用 A_i 加权、再 clip”的同一套逻辑。

7. Group Loss：一组 rollout 最后汇总成一个训练信号

实际实现会在 token、micro batch、mini batch 上求均值。教学上可以先看 response 级别：把 7 条 rollout 的 policy loss 平均。

group policy loss 0.00

这一组的完整链路

  1. 同题采样 7 条 response。
  2. 用规则 reward 判断最终答案是否正确。
  3. 在组内把 reward 标准化成 advantage。
  4. 用 old/new logprob 算 ratio，并用 0.2 clip 限制更新。
  5. 对每条 rollout 得到 policy loss，再做平均。
如果某条正确 rollout 的 probability 已经涨太快，clip 会让它少涨一点；如果错误 rollout 的 probability 被错误地涨高，loss 会强烈惩罚这个方向。

9. 附录：KL、Entropy、Mini-batch 放在哪里

主线已经讲完 policy loss。真实训练还会有一些工程项，它们不改变“rollout 到 loss”的主路径。

Ref KL

配置里 `use_kl_loss: True`、`kl_loss_coef: 0.001`。它约束当前 policy 不要离 reference policy 太远。

Entropy

训练脚本覆盖为 `actor_rollout_ref.actor.entropy_coeff=0`，所以这条主线里不展开 entropy bonus。

Mini-batch

`TRAIN_BS=32` 是 rollout prompt 批大小，`PPO_MINI_BS=8` 是更新批大小；实际 loss 会在更大的 batch 维度继续求均值。

一句话：GRPO 用同题多解的相对优劣替代 value model baseline，最后仍然通过 masked logprob 的 policy loss 更新模型。
